# api-gateway 格式转换修复方案

本文档基于 cc-switch 的实现，列出 api-gateway 当前格式转换功能与 cc-switch 的差异，并给出具体修复方案。

## 优先级说明

| 标记 | 含义 | 数量 |
|---|---|---|
| **P0 重要** | 影响功能正确性，不修会导致数据丢失或功能不可用 | 2 |
| **P1 优先** | 影响可靠性或某些场景下的正确性 | 5 |
| **P2 增强** | 提升健壮性，防御边缘场景 | 3 |
| **可选** | 与特定 upstream 提供商相关，通过配置开关控制 | 3 |

## 一、必须处理的问题

### 1. [P0] stream_options.include_usage 注入

**问题描述**
OpenAI Chat Completions 流式接口默认**不在最后一个 chunk 中返回 usage**。转换器依赖 usage 计算 Anthropic 的 `input_tokens` / `cache_read_input_tokens` / `output_tokens`。不注入 `stream_options.include_usage` 会导致转换后的 Anthropic 流 usage 全为 0，影响计费、监控和缓存统计。

**当前 api-gateway 实现**
- [`transform_stream.go:496`](src/transform_stream.go#L496) 读取 `chunk["usage"]`，但上游不返回时 `s.pendingUsage` 始终为 nil。
- 请求转换路径 [`transform.go:946 anthropicToOpenAIChatRequest`](src/transform.go#L946) 未注入 `stream_options`。

**cc-switch 的做法**
- [`transform.rs:241 inject_openai_stream_include_usage`](file:///d:/Files/GitHub/cc-switch/src-tauri/src/proxy/providers/transform.rs#L241)：在 `anthropic_to_openai` 时，若 `stream == true`，注入 `{"stream_options":{"include_usage":true}}`。

**修复方案**

文件：`src/transform.go`，函数 `anthropicToOpenAIChatRequest`（约 L946）

在 `result["stream"] = true` 之后，追加：

```go
// 强制注入 stream_options.include_usage，确保上游在末尾 chunk 返回 usage。
// 不注入则转换后的 Anthropic 流 input/output/cache tokens 全为 0。
if stream, _ := body["stream"].(bool); stream {
    if _, ok := result["stream_options"]; !ok {
        result["stream_options"] = map[string]interface{}{
            "include_usage": true,
        }
    } else if opts, ok := result["stream_options"].(map[string]interface{}); ok {
        if _, exists := opts["include_usage"]; !exists {
            opts["include_usage"] = true
        }
    }
}
```

注意：仅在 `anthropicToOpenAIChatRequest` 中注入。`anthropicToOpenAIResponsesRequest` 不需要（Responses API 默认返回 usage）。链式转换 `openai_chat → anthropic → gemini` 时，pivot 步骤的 `anthropicToGeminiRequest` 也不需要。

### 2. [P0] Gemini thoughtSignature 影子存储

**问题描述**
Gemini 的 `thoughtSignature` 字段（thinking 块的签名）**无法通过 Anthropic 格式往返传递**。Anthropic 的 `thinking` 块只有 `thinking` 文本和 `signature` 字段，结构与 Gemini 的 `thoughtSignature` 不兼容。当客户端进行多轮工具调用时：
1. 第一轮 Gemini 返回 thinking + thoughtSignature
2. 转成 Anthropic 格式给客户端，thoughtSignature 丢失
3. 客户端第二轮把上一轮的 assistant 消息（含 thinking）发回来
4. 转回 Gemini 时没有 thoughtSignature
5. **Gemini 拒绝请求**（签名校验失败）

**当前 api-gateway 实现**
- 完全没有 thoughtSignature 处理。Gemini 的 `thoughtSignature` 在 [`transform.go geminiToAnthropicResponse`](src/transform.go#L1864) 中被直接丢弃。
- 多轮工具调用场景下 Gemini 会 400。

**cc-switch 的做法**
- [`gemini_shadow.rs`](file:///d:/Files/GitHub/cc-switch/src-tauri/src/proxy/providers/gemini_shadow.rs)：`GeminiShadowStore` 按 `(provider_id, session_id)` 维度存储每个 assistant turn 的 `{text, thoughtSignature, toolCalls}`。
- 转换请求时 [`transform_gemini.rs:51 anthropic_to_gemini_with_shadow`](file:///d:/Files/GitHub/cc-switch/src-tauri/src/proxy/providers/transform_gemini.rs#L51) 从 shadow store 回放 thoughtSignature 到对应的 `parts`。
- 转换响应时 [`transform_gemini.rs:125 gemini_to_anthropic_with_shadow`](file:///d:/Files/GitHub/cc-switch/src-tauri/src/proxy/providers/transform_gemini.rs#L125) 将 thinking 块的 thoughtSignature 存入 shadow store。
- 流式响应 [`streaming_gemini.rs:194 build_shadow_assistant_parts`](file:///d:/Files/GitHub/cc-switch/src-tauri/src/proxy/providers/streaming_gemini.rs#L194) 在流结束时存入。
- 限制：默认 200 sessions，每 session 64 turns，LRU 淘汰。

**修复方案**

新增文件：`src/gemini_shadow.go`

```go
package main

import (
    "sync"
    "time"
)

// GeminiShadowKey 影子存储键（按 upstream + 会话维度隔离）
type GeminiShadowKey struct {
    UpstreamID string
    SessionID  string
}

// GeminiToolCallMeta 记录工具调用的签名信息
type GeminiToolCallMeta struct {
    ID             string
    Name           string
    Args            string
    ThoughtSignature string
}

// GeminiAssistantTurn 记录一个 assistant 轮次的 thinking + tool calls
type GeminiAssistantTurn struct {
    Text              string
    ThoughtSignature  string
    ToolCalls         []GeminiToolCallMeta
    CreatedAt         time.Time
}

// GeminiShadowStore 存储 Gemini thoughtSignature 以便多轮回放
type GeminiShadowStore struct {
    mu              sync.RWMutex
    maxSessions     int
    maxTurnsPerSess int
    sessions        map[GeminiShadowKey][]GeminiAssistantTurn
}

var defaultGeminiShadow = &GeminiShadowStore{
    maxSessions:     200,
    maxTurnsPerSess: 64,
    sessions:        make(map[GeminiShadowKey][]GeminiAssistantTurn),
}

// RecordTurn 记录一个 assistant turn
func (s *GeminiShadowStore) RecordTurn(key GeminiShadowKey, turn GeminiAssistantTurn) {
    s.mu.Lock()
    defer s.mu.Unlock()
    turn.CreatedAt = time.Now()
    turns := s.sessions[key]
    turns = append(turns, turn)
    // 超出每 session 上限 → 丢弃最老的
    if len(turns) > s.maxTurnsPerSess {
        turns = turns[len(turns)-s.maxTurnsPerSess:]
    }
    s.sessions[key] = turns
    // 超出总 session 上限 → LRU 淘汰
    if len(s.sessions) > s.maxSessions {
        var oldestKey GeminiShadowKey
        var oldestTime time.Time
        for k, t := range s.sessions {
            if len(t) > 0 && (oldestTime.IsZero() || t[0].CreatedAt.Before(oldestTime)) {
                oldestKey = k
                oldestTime = t[0].CreatedAt
            }
        }
        delete(s.sessions, oldestKey)
    }
}

// GetTurns 获取会话的所有历史 turns（用于回放 thoughtSignature）
func (s *GeminiShadowStore) GetTurns(key GeminiShadowKey) []GeminiAssistantTurn {
    s.mu.RLock()
    defer s.mu.RUnlock()
    return s.sessions[key]
}

// ClearSession 清除指定会话（请求失败时调用）
func (s *GeminiShadowStore) ClearSession(key GeminiShadowKey) {
    s.mu.Lock()
    defer s.mu.Unlock()
    delete(s.sessions, key)
}
```

集成点：

1. **SessionID 识别**：在 `gateway.go` 中，从请求 header 或 body 中提取会话标识。优先级：`x-session-id` header > `body["metadata"]["session_id"]` > 哈希(system prompt 前缀 + 第一条 user message)。fallback 用哈希保证无显式 session 时也能工作。

2. **请求转换** (`transform.go anthropicToGeminiRequest`)：增加 `shadowKey` 参数，转换 messages 时从 shadow store 取出对应 turn 的 thoughtSignature，注入到 Gemini `parts` 的 `thought: true` 块中。

3. **响应转换** (`transform.go geminiToAnthropicResponse`)：增加 `shadowKey` 参数，解析 Gemini 响应中的 `thoughtSignature`，存入 shadow store。

4. **流式响应** (`transform_stream.go geminiToAnthropicStream`)：在 `emitStop` 时，将累积的 `accumulatedText`、thinking signature、tool calls 存入 shadow store。

5. **失败清理**：在 `gateway.go` 请求失败 `continue` 到下一个 upstream 时，调用 `ClearSession` 避免脏数据。

注意：此修复涉及 `TransformRequest` / `TransformResponse` / `NewStreamConverter` 的签名变更（需传入 shadowKey）。建议增加 `TransformContext` 结构体封装这些上下文：

```go
type TransformContext struct {
    InFormat   string
    OutFormat  string
    Model      string
    IsStream   bool
    ShadowKey  GeminiShadowKey // 仅 Gemini 方向使用
}
```

### 3. [P1] redacted_thinking 占位符

**问题描述**
Anthropic 的 `redacted_thinking` 块表示加密的 thinking 内容（模型不暴露明文）。当前 api-gateway 在转 OpenAI/Gemini 时**直接丢弃**，导致：
- 客户端看不到任何提示，以为模型没有 thinking
- 多轮对话中后续轮次缺失 thinking 上下文，模型行为异常

**当前 api-gateway 实现**
- [`transform.go:1129`](src/transform.go#L1129) (Anthropic→OpenAI Chat)：`case "thinking": // thinking blocks are not forwarded to OpenAI`，无 `redacted_thinking` case。
- [`transform.go:1414`](src/transform.go#L1414) (Anthropic→OpenAI Responses)：同上。
- Anthropic→Gemini 方向也未处理。

**cc-switch 的做法**
- 转成 `[redacted thinking]` 占位字符串，保留在 `reasoning_content` 或对应位置。
- 常量 `ANTHROPIC_REDACTED_THINKING_PLACEHOLDER = "[redacted thinking]"`（[`claude.rs`](file:///d:/Files/GitHub/cc-switch/src-tauri/src/proxy/providers/claude.rs)）。

**修复方案**

文件：`src/transform.go`

**步骤 1**：定义常量

```go
const redactedThinkingPlaceholder = "[redacted thinking]"
```

**步骤 2**：在所有 Anthropic→X 的消息转换中，新增 `redacted_thinking` case。

Anthropic→OpenAI Chat（[`transform.go:1129`](src/transform.go#L1129) 附近）：

```go
case "thinking":
    // thinking blocks are not forwarded to OpenAI
case "redacted_thinking":
    // 加密的 thinking 无法还原，用占位符保留语义
    textParts = append(textParts, redactedThinkingPlaceholder)
```

Anthropic→OpenAI Responses（[`transform.go:1414`](src/transform.go#L1414) 附近）：同上追加 case。

Anthropic→Gemini（搜索 `transform.go` 中 `anthropicToGeminiRequest` 内的 content 转换，约 L1550 附近）：

```go
case "redacted_thinking":
    parts = append(parts, map[string]interface{}{
        "text":   redactedThinkingPlaceholder,
        "thought": true,
    })
```

**步骤 3**：反向转换（X→Anthropic）中，若发现占位符文本，无需还原（信息已丢失，占位符作为普通 text 传递即可）。

### 4. [P1] cache_control 自动注入

**问题描述**
Anthropic Prompt Caching 需要在 `tools`、`system`、消息内容块上显式设置 `cache_control: {"type":"ephemeral"}` 断点。客户端不设置时，所有请求都全量计费，无法享受缓存折扣。cc-switch 支持自动注入断点；api-gateway 完全不支持。

**当前 api-gateway 实现**
- 只在 OpenAI/Responses 方向**剥离** cache_control（因为 OpenAI 不认）。
- 无任何自动注入逻辑。

**cc-switch 的做法**
- [`cache_injector.rs`](file:///d:/Files/GitHub/cc-switch/src-tauri/src/proxy/providers/cache_injector.rs)：`inject(body, config)` 在以下位置注入最多 4 个断点：
  1. `tools` 数组末尾（最后一个 tool 上）
  2. `system` 末尾（字符串转数组后，最后一个 block 上）
  3. 最后一条 assistant 消息的最后一个非 thinking block
  4. （可选）最后一条 user 消息末尾
- 已有断点升级 TTL（`5m` → 无 ttl 字段；其他 → `{"type":"ephemeral","ttl":"<ttl>"}`）。

**修复方案**

新增文件：`src/cache_injector.go`

```go
package main

import "encoding/json"

// CacheInjectorConfig 控制自动 cache_control 断点注入
type CacheInjectorConfig struct {
    Enabled bool   `json:"enabled"`     // 是否启用
    TTL     string `json:"ttl,omitempty"` // "5m" 或 "1h"，空=默认 5m
}

// injectCacheControl 在 Anthropic 请求体的关键位置注入 cache_control 断点。
// 最多注入 4 个断点（Anthropic 上限）：
//   1. tools 数组末尾
//   2. system 末尾
//   3. 最后一条 assistant 消息的最后一个非 thinking block
//   4. 最后一条 user 消息末尾
func injectCacheControl(body map[string]interface{}, cfg CacheInjectorConfig) {
    if !cfg.Enabled {
        return
    }
    ttl := cfg.TTL
    var cacheControl interface{}
    if ttl == "" || ttl == "5m" {
        cacheControl = map[string]interface{}{"type": "ephemeral"}
    } else {
        cacheControl = map[string]interface{}{"type": "ephemeral", "ttl": ttl}
    }

    remaining := 4

    // 1. tools 末尾
    if remaining > 0 {
        if tools, ok := body["tools"].([]interface{}); ok && len(tools) > 0 {
            if lastTool, ok := tools[len(tools)-1].(map[string]interface{}); ok {
                if _, exists := lastTool["cache_control"]; !exists {
                    lastTool["cache_control"] = cacheControl
                    remaining--
                }
            }
        }
    }

    // 2. system 末尾
    if remaining > 0 {
        injectCacheControlToSystem(body, cacheControl, &remaining)
    }

    // 3 & 4. 消息中的最后 assistant / user
    if remaining > 0 {
        injectCacheControlToMessages(body, cacheControl, &remaining)
    }
}

func injectCacheControlToSystem(body map[string]interface{}, cc interface{}, remaining *int) {
    sys, ok := body["system"]
    if !ok {
        return
    }
    switch s := sys.(type) {
    case string:
        // 字符串 → 转为数组以附加 cache_control
        body["system"] = []interface{}{
            map[string]interface{}{"type": "text", "text": s, "cache_control": cc},
        }
        *remaining--
    case []interface{}:
        if len(s) > 0 {
            if last, ok := s[len(s)-1].(map[string]interface{}); ok {
                if _, exists := last["cache_control"]; !exists {
                    last["cache_control"] = cc
                    *remaining--
                }
            }
        }
    }
}

func injectCacheControlToMessages(body map[string]interface{}, cc interface{}, remaining *int) {
    messages, ok := body["messages"].([]interface{})
    if !ok || len(messages) == 0 {
        return
    }
    // 从后往前找 assistant 和 user
    for i := len(messages) - 1; i >= 0 && *remaining > 0; i-- {
        msg, ok := messages[i].(map[string]interface{})
        if !ok {
            continue
        }
        role, _ := msg["role"].(string)
        if role != "assistant" && role != "user" {
            continue
        }
        content, ok := msg["content"].([]interface{})
        if !ok || len(content) == 0 {
            continue
        }
        // 找最后一个非 thinking block
        for j := len(content) - 1; j >= 0; j-- {
            block, ok := content[j].(map[string]interface{})
            if !ok {
                continue
            }
            t, _ := block["type"].(string)
            if t == "thinking" || t == "redacted_thinking" {
                continue
            }
            if _, exists := block["cache_control"]; !exists {
                block["cache_control"] = cc
                *remaining--
            }
            break
        }
    }
}
```

**集成点**：在 `gateway.go` 请求转发前，若 `inFormat == formatAnthropic && outFormat == formatAnthropic`（透传 Anthropic 上游）或 `outFormat == formatAnthropic`（X→Anthropic 转换后），调用 `injectCacheControl`。

配置：在 `UpstreamConfig` 中新增字段（见本文档末尾"配置变更汇总"）。

### 5. [P1] 重复 finish_reason 去重

**问题描述**
某些 OpenAI 兼容上游（如 OpenRouter、部分自建网关）会在流式响应中**多次发送 `finish_reason`**。当前 api-gateway：
- OpenAI Chat 方向有去重（`hasEmittedDelta` 字段）✓
- OpenAI Responses 方向**无去重**
- Gemini 方向**无去重**

虽然 Responses/Gemini 按规范只发一次终止事件，但防御性去重能避免异常上游导致 `pendingStopReason` 被覆盖。

**当前 api-gateway 实现**
- [`transform_stream.go:489`](src/transform_stream.go#L489) OpenAI Chat：`if fr, ok := ...; ok && fr != "" && !s.hasEmittedDelta` ✓ 已去重
- [`transform_stream.go:812`](src/transform_stream.go#L812) Responses：`s.pendingStopReason = mapResponsesStopReason(...)` 无守卫 ✗
- [`transform_stream.go:1064`](src/transform_stream.go#L1064) Gemini：`s.pendingStopReason = mapGeminiFinishReason(...)` 无守卫 ✗

**cc-switch 的做法**
- [`streaming.rs`](file:///d:/Files/GitHub/cc-switch/src-tauri/src/proxy/providers/streaming.rs) `pending_message_delta` 字段 + `has_pending_message_delta` 守卫，首次设置后不再覆盖。

**修复方案**

文件：`src/transform_stream.go`

**步骤 1**：在 `openaiResponsesToAnthropicStream`（L567）和 `geminiToAnthropicStream`（L879）结构体中新增字段：

```go
hasSetStopReason bool  // 防止重复 finish_reason 覆盖 stop_reason
```

**步骤 2**：Responses 方向（L812 附近）：

```go
case "response.completed":
    resp := getMap(data, "response")
    if resp != nil {
        if !s.hasSetStopReason {
            s.hasSetStopReason = true
            status := getString(resp, "status")
            hasToolUse := len(s.toolBlocks) > 0
            s.pendingStopReason = mapResponsesStopReason(status, hasToolUse)
        }
        if usage := getMap(resp, "usage"); usage != nil {
            s.pendingUsage = buildAnthropicUsageFromResponses(usage)
        }
    }
    output = append(output, s.emitStop()...)

case "response.incomplete":
    if !s.hasSetStopReason {
        s.hasSetStopReason = true
        s.pendingStopReason = "max_tokens"
    }
    // ... 后续 usage 逻辑不变
```

**步骤 3**：Gemini 方向（L1064 附近）：

```go
if fr := getString(candidate, "finishReason"); fr != "" {
    if !s.hasSetStopReason {
        s.hasSetStopReason = true
        s.pendingStopReason = mapGeminiFinishReason(fr, hasToolUse)
    }
    if usage := getMap(data, "usageMetadata"); usage != nil {
        s.pendingUsage = buildAnthropicUsageFromGemini(usage)
    }
    output = append(output, s.emitStop()...)
}
```

**步骤 4**（可选统一）：考虑将 `hasEmittedDelta`（OpenAI Chat 方向）重命名为 `hasSetStopReason` 以保持命名一致，但这是 cosmetic 改动，不影响功能。

### 6. [P1] 懒发送 message_start

**问题描述**
当前 api-gateway 在**收到第一个 chunk 时立即发送 `message_start`**，即使该 chunk 不含任何内容（如 OpenAI 的首个 `role` delta）。如果上游返回空响应或立即出错，客户端会收到一个 `message_start` 后无任何内容，形成"悬挂消息"。

**当前 api-gateway 实现**
- [`transform_stream.go:300-313`](src/transform_stream.go#L300) OpenAI Chat 方向：首个 chunk 即发 `message_start`，usage 硬编码 `{"input_tokens":0,"output_tokens":0}`。
- [`transform_stream.go:673-685`](src/transform_stream.go#L673) Responses 方向：同上。
- [`transform_stream.go:937-953`](src/transform_stream.go#L937) Gemini 方向：同上。

**cc-switch 的做法**
- 懒发送：`message_start` 推迟到**首个实际内容 delta 或 usage chunk** 时才发送。
- 若上游空响应，不发送 `message_start`，客户端直接收到错误或空。

**修复方案**

文件：`src/transform_stream.go`

对三个 `xToAnthropicStream` 转换器统一改造。以 `openaiChatToAnthropicStream` 为例：

**步骤 1**：新增 `emitMessageStart` 方法，将 `message_start` 发送逻辑提取出来：

```go
func (s *openaiChatToAnthropicStream) emitMessageStart() []byte {
    s.hasSentMessageStart = true
    event := map[string]interface{}{
        "type": "message_start",
        "message": map[string]interface{}{
            "id":    s.messageID,
            "type":  "message",
            "role":  "assistant",
            "model": s.model,
            "usage": map[string]interface{}{"input_tokens": 0, "output_tokens": 0},
        },
    }
    return sseEvent("message_start", mustJSON(event))
}
```

**步骤 2**：新增 `ensureMessageStart` helper，在有实际内容时才发送：

```go
func (s *openaiChatToAnthropicStream) ensureMessageStart(output *[]byte) {
    if !s.hasSentMessageStart {
        *output = append(*output, s.emitMessageStart()...)
    }
}
```

**步骤 3**：删除首个 chunk 的无条件发送（L300-313），改为在以下时机调用 `ensureMessageStart`：
- 收到首个 `text_delta`（content 实际内容）时
- 收到首个 `tool_calls` delta 时
- 收到 `usage` 且 `pendingUsage == nil`（首个 usage chunk）时
- 收到 `finish_reason` 时（兜底，确保至少有 message_start）
- `handleDone` / `emitStop` 时（兜底）

**步骤 4**：Responses 和 Gemini 方向同样改造。

**注意事项**：
- 懒发送后，`message_start` 中的 `usage` 仍为 0（Anthropic 规范要求 message_start 必须含 usage）。真实的 input_tokens 在 `message_delta` 中通过 `pendingUsage` 发送。
- `emitStop` 必须调用 `ensureMessageStart` 作为兜底，否则空响应会导致只有 `message_stop` 没有 `message_start`，客户端解析报错。

### 7. [P1] UTF-8 安全累积

**问题描述**
TCP 流式传输中，一个多字节 UTF-8 字符可能被拆分到两个 chunk（如中文字符、emoji）。当前 api-gateway 的 `takeSSEBlock` 直接按字节切分，虽然 SSE 事件以 `\n\n` 分隔通常不会拆分字符，但：
- 若上游在 SSE 事件**中间**断开 chunk（非 `\n\n` 边界），且断点恰好在多字节字符中间，`string(buf[:bestPos])` 会产生无效 UTF-8。
- 这会损坏工具调用的 JSON 参数（`input_json_delta` 的 `partial_json`），导致客户端 JSON 解析失败。

**当前 api-gateway 实现**
- [`transform_stream.go:31 takeSSEBlock`](src/transform_stream.go#L31)：直接 `string(buf[:bestPos])`，无 UTF-8 安全处理。
- grep 确认整个 `transform_stream.go` 无 `utf8` / `rune` / `InvalidByte` 引用。

**cc-switch 的做法**
- 所有流式转换器使用 `append_utf8_safe` / `utf8_remainder` 处理 chunk 边界：
  - `append_utf8_safe(buf, chunk)`：只追加完整的 UTF-8 字符，返回可能的不完整尾部。
  - `utf8_remainder(chunk)`：提取最后一个不完整字符的字节，留到下次拼接。

**修复方案**

文件：`src/transform_stream.go`

**步骤 1**：新增 UTF-8 安全追加函数

```go
import "unicode/utf8"

// appendUTF8Safe 将 chunk 追加到 buffer，只保留完整的 UTF-8 字符。
// 返回尾部不完整字节（留待下次拼接）。
// 注意：SSE 事件以 \n\n 分隔，\n 是单字节 ASCII，不会出现在多字节字符内部，
// 因此 takeSSEBlock 的分隔符查找不受影响。此函数仅保护 buffer.String() 的有效性。
func appendUTF8Safe(buf *bytes.Buffer, chunk []byte) []byte {
    if len(chunk) == 0 {
        return nil
    }
    // 找到最后一个完整 UTF-8 字符的边界
    valid := len(chunk)
    for valid > 0 && !utf8.Valid(chunk[:valid]) {
        valid--
    }
    if valid > 0 {
        buf.Write(chunk[:valid])
    }
    return chunk[valid:] // 返回不完整尾部
}
```

**步骤 2**：修改各转换器的 `Feed` 方法

以 `openaiChatToAnthropicStream.Feed`（L227）为例：

```go
func (s *openaiChatToAnthropicStream) Feed(chunk []byte) ([]byte, error) {
    // 先处理上次残留的不完整 UTF-8 尾部
    if len(s.utf8Remainder) > 0 {
        chunk = append(s.utf8Remainder, chunk...)
        s.utf8Remainder = nil
    }
    s.utf8Remainder = appendUTF8Safe(&s.buffer, chunk)
    // ... 后续 takeSSEBlock 逻辑不变
}
```

**步骤 3**：在结构体中新增 `utf8Remainder []byte` 字段。

**步骤 4**：`Flush` 方法中，若有残留 `utf8Remainder`，尝试作为最后一个不完整事件处理（通常丢弃，因为不完整事件无法解析）。

**影响范围**：6 个流式转换器 + `chainedStreamConverter`。链式转换器的 `first` 输出已经是完整 SSE 事件，`second` 不需要再处理 UTF-8 边界，但为安全起见也加上。

### 8. [P2] thinking signature 自动修复

**问题描述**
Anthropic 的 `thinking` 块含 `signature` 字段，用于验证 thinking 完整性。客户端修改请求（如删除某些 thinking 块、插入新消息）会导致签名校验失败，上游返回 400。cc-switch 能自动检测并修复；api-gateway 无此能力，直接返回 400 给客户端。

**当前 api-gateway 实现**
- 无任何 thinking signature 修复逻辑。

**cc-switch 的做法**
- [`thinking_rectifier.rs`](file:///d:/Files/GitHub/cc-switch/src-tauri/src/proxy/providers/thinking_rectifier.rs)（430+ 行）：
  - `should_rectify_thinking_signature`：检测 7+ 种错误消息模式（"signature is not valid"、"must start with a thinking block"、"Expected thinking or redacted_thinking but found tool_use" 等）
  - `rectify_anthropic_request`：移除 thinking/redacted_thinking 块、剥离非 thinking 块的 signature 字段、条件性移除顶层 thinking 配置
  - 触发后自动重试请求

**修复方案**

新增文件：`src/thinking_rectifier.go`

```go
package main

import (
    "regexp"
    "strings"
)

// thinkingSignatureErrorPatterns 匹配 Anthropic 返回的 thinking signature 错误
var thinkingSignatureErrorPatterns = []*regexp.Regexp{
    regexp.MustCompile(`(?i)signature.*not.*valid`),
    regexp.MustCompile(`(?i)thought signature is not valid`),
    regexp.MustCompile(`(?i)must start with a thinking block`),
    regexp.MustCompile(`(?i)expected thinking or redacted_thinking but found`),
    regexp.MustCompile(`(?i)signature.*field required`),
    regexp.MustCompile(`(?i)extra inputs are not permitted`),
    regexp.MustCompile(`(?i)thinking.*cannot be modified`),
    regexp.MustCompile(`(?i)(illegal|invalid) request`),
}

// shouldRectifyThinkingSignature 检测错误响应是否为 thinking signature 问题
func shouldRectifyThinkingSignature(errorBody []byte) bool {
    bodyStr := string(errorBody)
    for _, p := range thinkingSignatureErrorPatterns {
        if p.MatchString(bodyStr) {
            return true
        }
    }
    return false
}

// rectifyAnthropicRequest 清理请求中的 thinking 块以通过签名校验
func rectifyAnthropicRequest(body map[string]interface{}) {
    messages, ok := body["messages"].([]interface{})
    if !ok {
        return
    }

    lastAssistantIdx := -1
    for i := len(messages) - 1; i >= 0; i-- {
        if msg, ok := messages[i].(map[string]interface{}); ok {
            if role, _ := msg["role"].(string); role == "assistant" {
                lastAssistantIdx = i
                break
            }
        }
    }

    for i, msgRaw := range messages {
        msg, ok := msgRaw.(map[string]interface{})
        if !ok {
            continue
        }
        content, ok := msg["content"].([]interface{})
        if !ok {
            continue
        }

        // 移除所有 thinking / redacted_thinking 块
        var newContent []interface{}
        for _, blockRaw := range content {
            block, ok := blockRaw.(map[string]interface{})
            if !ok {
                newContent = append(newContent, blockRaw)
                continue
            }
            t, _ := block["type"].(string)
            if t == "thinking" || t == "redacted_thinking" {
                continue // 移除
            }
            // 剥离非 thinking 块上残留的 signature 字段
            delete(block, "signature")
            newContent = append(newContent, block)
        }
        msg["content"] = newContent
    }

    // 如果最后一条 assistant 消息移除 thinking 后不以 thinking 开头，
    // 且顶层 thinking 配置为 enabled，则移除 thinking 配置
    // （Anthropic 要求最后一条 assistant 消息必须以 thinking 开头才能启用 extended thinking）
    if lastAssistantIdx >= 0 {
        if thinking, ok := body["thinking"].(map[string]interface{}); ok {
            if t, _ := thinking["type"].(string); t == "enabled" {
                delete(body, "thinking")
            }
        }
    }
}
```

**集成点**：在 `gateway.go` 的请求重试循环中，当上游返回 400 且 `shouldRectifyThinkingSignature(body)` 为 true 时：
1. 调用 `rectifyAnthropicRequest(bodyBytes)`
2. 重新序列化 body
3. 重试同一 upstream（最多重试 1 次）

```go
// gateway.go 伪代码
if resp.StatusCode == 400 && inFormat == formatAnthropic {
    if shouldRectifyThinkingSignature(rawBody) && !rectified {
        rectifyAnthropicRequest(bodyMap)
        bodyBytes, _ = json.Marshal(bodyMap)
        rectified = true
        // 重新发起请求到同一 upstream
        continue // 或 goto retry
    }
}
```

### 9. [P2] 无限空白中止

**问题描述**
某些上游（如 GitHub Copilot）在工具调用参数流中存在 bug：无限输出空白字符。当前 api-gateway 无限累积这些空白，导致内存增长和客户端超时。

**当前 api-gateway 实现**
- [`transform_stream.go`](src/transform_stream.go) `toolBlockState` 无连续空白计数，无限累积 `pendingArgs`。

**cc-switch 的做法**
- [`streaming.rs:100`](file:///d:/Files/GitHub/cc-switch/src-tauri/src/proxy/providers/streaming.rs#L100) `INFINITE_WHITESPACE_THRESHOLD = 500`
- `ToolBlockState` 含 `consecutive_whitespace` 计数器
- 超过 500 连续空白 → `aborted = true`，丢弃后续 delta，发送空的 `input_json_delta` 关闭 block

**修复方案**

文件：`src/transform_stream.go`

**步骤 1**：定义阈值常量

```go
const infiniteWhitespaceThreshold = 500
```

**步骤 2**：在 `toolBlockState` 结构体（约 L870 附近）新增字段：

```go
type toolBlockState struct {
    // ... 现有字段
    consecutiveWhitespace int  // 连续空白计数（防御 Copilot bug）
    aborted               bool // 是否已因无限空白中止
}
```

**步骤 3**：在 `input_json_delta` 处理逻辑中（OpenAI Chat 方向约 L460 附近），追加空白检测：

```go
// 处理 tool_calls delta
if args != "" {
    if block.aborted {
        continue // 已中止，丢弃
    }
    // 检测无限空白
    allWhitespace := true
    for _, r := range args {
        if r != ' ' && r != '\t' && r != '\n' && r != '\r' {
            allWhitespace = false
            break
        }
    }
    if allWhitespace {
        block.consecutiveWhitespace += len(args)
        if block.consecutiveWhitespace > infiniteWhitespaceThreshold {
            block.aborted = true
            // 发送空 partial_json 关闭 block
            // 记录日志
            continue
        }
    } else {
        block.consecutiveWhitespace = 0
    }
    block.pendingArgs += args
    // ... 正常发送 input_json_delta
}
```

**步骤 4**：在 `handleDone` / `emitStop` 中，若 `block.aborted`，发送 `content_block_stop` 但不发送完整的 tool_use（或发送空 input）。

**影响范围**：主要影响 `openaiChatToAnthropicStream`。Responses 和 Gemini 方向若也有类似累积逻辑，同样处理。

### 10. [P2] reasoning effort 4 档映射

**问题描述**
当前 api-gateway 的 reasoning effort 映射只有 3 档（low/medium/high），且阈值与 cc-switch 不同。cc-switch 额外支持 `xhigh`（对应 `adaptive` budget）和更细分的阈值。

**当前 api-gateway 实现**
- [`transform.go:298 resolveReasoningEffort`](src/transform.go#L298)：
  - `budget >= 32000` → high
  - `budget >= 16000` → medium
  - `< 16000` → low
  - 无 budget → medium
- 反向 [`transform.go:323`](src/transform.go#L323)：low→8000, medium→16000, high→32000
- 不支持 `adaptive` / `xhigh`。
- 不读取 `output_config.effort`。

**cc-switch 的做法**
- [`transform.rs:82 resolve_reasoning_effort`](file:///d:/Files/GitHub/cc-switch/src-tauri/src/proxy/providers/transform.rs#L82)：
  - 优先级 1：`output_config.effort`（Anthropic 风格），`max` → `xhigh`
  - 优先级 2：`thinking.budget_tokens`：
    - `adaptive` → `xhigh`
    - `< 4096` → `low`
    - `4096 ≤ budget < 16384` → `medium`
    - `≥ 16384` → `high`
- 更细分的阈值，且支持 `xhigh` 档位。

**修复方案**

文件：`src/transform.go`

**步骤 1**：修改 `resolveReasoningEffort`（L298）：

```go
func resolveReasoningEffort(body map[string]interface{}) string {
    // 优先级 1：output_config.effort（Anthropic 风格）
    if oc := getMap(body, "output_config"); oc != nil {
        if e, ok := asString(oc["effort"]); ok && e != "" {
            if e == "max" {
                return "xhigh"
            }
            return e
        }
    }

    // 优先级 2：thinking.budget_tokens
    thinking := getMap(body, "thinking")
    if thinking == nil {
        return ""
    }
    t, ok := asString(thinking["type"])
    if !ok || t != "enabled" {
        return ""
    }
    // adaptive 模式
    if a, ok := asString(thinking["budget_tokens"]); ok && a == "adaptive" {
        return "xhigh"
    }
    if budget, ok := asFloat64(thinking["budget_tokens"]); ok {
        if budget >= 16384 {
            return "high"
        }
        if budget >= 4096 {
            return "medium"
        }
        return "low"
    }
    return "medium"
}
```

**步骤 2**：修改反向映射 `resolveAnthropicThinkingFromReasoning`（L323），增加 `xhigh`：
