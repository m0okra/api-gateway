## 一、P0 — 影响功能正确性

### 1. [P0] 缺少 OpenAI Chat ↔ OpenAI Responses 转换方向

**问题描述**
api-gateway 把 `openai_chat` 与 `openai_responses` 视作同一类（`isOpenAIVariant`），两者之间不做转换直接 passthrough。但两者请求/响应结构完全不同（Chat 用 `messages`/`choices`，Responses 用 `input`/`output`）。当 Responses 客户端打到只支持 Chat Completions 的 upstream（或反向），上游会因字段不匹配返回 400。

**当前 api-gateway 实现**
- [transform.go:79-82](file:///d:/Files/api-gateway/src/transform.go#L79-L82) `needsTransform`：`isOpenAIVariant(inFormat) && isOpenAIVariant(outFormat)` 直接返回 false。
- [transform.go:85-87](file:///d:/Files/api-gateway/src/transform.go#L85-L87) `isOpenAIVariant`：`openai_chat` 与 `openai_responses` 都算 OpenAI 变体。

**cc-switch 的做法**
- [transform_codex_chat.rs](file:///d:/Files/GitHub/cc-switch/src-tauri/src/proxy/providers/transform_codex_chat.rs)（116KB）+ [streaming_codex_chat.rs](file:///d:/Files/GitHub/cc-switch/src-tauri/src/proxy/providers/streaming_codex_chat.rs)（51KB）专门处理 Chat ↔ Responses 双向转换。
- 支持普通 function tool、namespace tool、custom tool、tool_search 等多种工具形态。

**修复方案**

文件：`src/transform.go`、`src/transform_stream.go`

**步骤 1**：修改 `needsTransform`，移除 `isOpenAIVariant` 短路：

```go
func needsTransform(inFormat, outFormat string) bool {
    if inFormat == formatUnknown || outFormat == formatUnknown {
        return false
    }
    if inFormat == outFormat {
        return false
    }
    // 不再把 openai_chat 与 openai_responses 视作同一格式
    return true
}
```

**步骤 2**：在 `TransformRequest` 的 switch 中新增两个方向：

```go
case formatOpenAIChat == inFormat && formatOpenAIResponses == outFormat:
    return openaiChatToResponsesRequest(body)
case formatOpenAIResponses == inFormat && formatOpenAIChat == outFormat:
    return openaiResponsesToChatRequest(body)
```

**步骤 3**：实现 `openaiChatToResponsesRequest`：
- `messages` → `input`：user/assistant 消息转 `{"type":"message","role":...,"content":[{"type":"input_text"/"output_text","text":...}]}`；assistant 的 `tool_calls` 转 `{"type":"function_call","call_id":...,"name":...,"arguments":...}`；user 的 `tool` role 消息（tool 结果）转 `{"type":"function_call_output","call_id":...,"output":...}`。
- `max_tokens` → `max_output_tokens`
- `stop` → `stop`（数组）
- `tools` 中 `function` 类型工具转 Responses 的 `{"type":"function","name":...,"description":...,"parameters":...}`。
- `tool_choice` 按 Responses 格式映射。
- `reasoning_effort` → `{"reasoning":{"effort":...}}`。

**步骤 4**：实现 `openaiResponsesToChatRequest`：反向映射，`input` → `messages`，`instructions` → `system`，`max_output_tokens` → `max_tokens`，等。

**步骤 5**：流式同理，在 `NewStreamConverter` 中新增两个方向，实现 `chatToResponsesStream` 和 `responsesToChatStream`，把 Anthropic 中间形态改为目标格式的事件流。

**工作量提示**：这是本批最大的改动。建议参考 cc-switch 的 `transform_codex_chat.rs` 和 `streaming_codex_chat.rs` 完整 port。可分阶段：先非流式，后流式。

### 2. [P0] OpenAI Chat → Anthropic 流式丢弃 reasoning_content

**问题描述**
DeepSeek、Kimi、Moonshot、OpenRouter（o1/o3）等上游在流式响应的 `delta` 中通过 `reasoning` 或 `reasoning_content` 字段返回 thinking 内容。api-gateway 的转换器只读 `delta.content`/`delta.tool_calls`/`delta.function_call`，完全忽略 reasoning 字段，导致 thinking 被静默丢弃，客户端看不到模型的推理过程。

**当前 api-gateway 实现**

**修复方案**

文件：`src/transform_stream.go`，函数 `openaiChatToAnthropicStream.Feed`（L390 附近）

在 `delta.content` 处理之前插入 reasoning 处理分支：

```go

if r, ok := asString(delta["reasoning"]); ok && r != "" {
    reasoning = r
} else if r, ok := asString(delta["reasoning_content"]); ok && r != "" {
    reasoning = r
}
if reasoning != "" {
    s.ensureMessageStart(&output)
```

**步骤 4**：在 `emitStop` / `handleDone` 中，若 `openThinkingBlockIndex != nil`，发 `content_block_stop` 关闭。

### 5. [P0] 缺少 Codex OAuth 后端协议整形

**问题描述**
当 upstream 是 ChatGPT Plus/Pro 的 Codex 后端（`chatgpt.com/backend-api/codex`）时，该后端对 Responses API 请求有特殊要求：必须 `store: false`、必须 `include: ["reasoning.encrypted_content"]`、`service_tier: "priority"`（fast 模式）、必须 `stream: true`，且拒绝 `max_output_tokens`/`temperature`/`top_p` 参数。api-gateway 不做任何整形，直接转发会导致 400 "Unsupported parameter"。

**当前 api-gateway 实现**
- [transform.go:1320-1401](file:///d:/Files/api-gateway/src/transform.go#L1320-L1401) `anthropicToOpenAIResponsesRequest`：无 Codex 后端检测，无字段注入/剥离。

**cc-switch 的做法**
- [transform_responses.rs:147-203](file:///d:/Files/GitHub/cc-switch/src-tauri/src/proxy/providers/transform_responses.rs#L147-L203)：检测 base URL 含 `chatgpt.com/backend-api/codex` 时：
  - 注入 `store: false`
  - 注入 `include: ["reasoning.encrypted_content"]`
  - 注入 `service_tier: "priority"`（若配置 fast 模式）
  - 强制 `stream: true`
  - 删除 `max_output_tokens`、`temperature`、`top_p`

**修复方案**

文件：`src/transform.go`

**步骤 1**：新增检测函数：

```go
// isCodexOAuthBackend 判断 upstream 是否为 ChatGPT Codex OAuth 后端。
// 该后端对 Responses API 请求有特殊字段要求。
func isCodexOAuthBackend(baseURL string) bool {
    lower := strings.ToLower(baseURL)
    return strings.Contains(lower, "chatgpt.com/backend-api/codex")
}
```

**步骤 2**：修改 `anthropicToOpenAIResponsesRequest` 签名，接收 baseURL（或在 `TransformRequest` 中后处理）。推荐后处理，避免签名变更蔓延：

```go
// 在 TransformRequest 中，转换完成后：
if outFormat == formatOpenAIResponses {
    if cfg, ok := ctx.Value(upstreamBaseURLKey).(string); ok && isCodexOAuthBackend(cfg) {
        shapeForCodexBackend(result)
    }
}
```

**步骤 3**：实现 `shapeForCodexBackend`：

```go
func shapeForCodexBackend(body map[string]interface{}) {
    body["store"] = false
    body["include"] = []interface{}{"reasoning.encrypted_content"}
    if _, ok := body["service_tier"]; !ok {
        body["service_tier"] = "priority"
    }
    body["stream"] = true
    // Codex 后端拒绝以下参数
    delete(body, "max_output_tokens")
    delete(body, "temperature")
    delete(body, "top_p")
}
```

**步骤 4**：配置层无需改动，通过 upstream 的 `targetBase` 自动检测。

**注意**：此修复需要 `TransformRequest` 能拿到 upstream 的 `targetBase`。若当前签名不支持，可通过 `TransformContext` 结构体传递（参见第一轮文档 #2 的 TransformContext 提议）。

### 6. [P0] 流式上游出错时不发 error 事件

**问题描述**
当 upstream 在流式传输过程中返回非 EOF 错误（连接中断、上游 500 等），api-gateway 把错误透传到 HTTP 层，客户端的 SSE 流会突然中断且看不到任何 Anthropic 格式的 error 事件。更严重的是，`Flush()` 仍会发送合成的 `message_delta` + `message_stop`，把失败伪装成正常完成，客户端误以为响应成功但内容被截断。

**当前 api-gateway 实现**
- [transform_stream.go:2187-2235](file:///d:/Files/api-gateway/src/transform_stream.go#L2187-L2235) `transformStreamReader.Read`：非 EOF 错误存入 `pendingErr`，下次 Read 返回，未发任何 SSE error 事件。
- `Flush()` 无条件发送 `message_delta` + `message_stop`，不区分上游是否出错。

**cc-switch 的做法**
- [streaming.rs:627-647](file:///d:/Files/GitHub/cc-switch/src-tauri/src/proxy/providers/streaming.rs#L627-L647)：
  - 检测到错误时设 `stream_ended_with_error = true`。
  - 发送 `event: error\ndata: {"type":"error","error":{"type":"stream_error","message":...}}`。
  - `stream_ended_with_error` 为 true 时，跳过合成的 `message_delta`/`message_stop`，避免把失败伪装成成功。

**修复方案**

文件：`src/transform_stream.go`

**步骤 1**：在 `transformStreamReader` 结构体新增字段：

```go
type transformStreamReader struct {
    upstream  io.ReadCloser
    converter StreamConverter
    buf       bytes.Buffer
    tmpBuf    [32 * 1024]byte
    pendingErr error
    done       bool
    // 新增：标记上游是否以错误结束，抑制合成的成功终止事件
    streamEndedWithError bool
}
```

**步骤 2**：修改 `Read`，非 EOF 错误时先发 error 事件：

```go
if err != nil {
    if err == io.EOF {
        // ... 现有 EOF 逻辑不变
    }
    // 非 EOF 错误：发 Anthropic 格式 error 事件
    if !r.streamEndedWithError {
        r.streamEndedWithError = true
        errEvent := map[string]interface{}{
            "type": "error",
            "error": map[string]interface{}{
                "type":    "stream_error",
                "message": fmt.Sprintf("upstream stream error: %v", err),
            },
        }
        r.buf.Write(sseEvent("error", mustJSON(errEvent)))
    }
    r.pendingErr = err
    continue
}
```

**步骤 3**：修改 `Flush` 逻辑（或 converter 的 Flush），若 `streamEndedWithError` 则不发 `message_delta`/`message_stop`：

在 `transformStreamReader` 层面，可在 EOF 分支的 `Flush()` 调用前判断。但 `Flush()` 是 converter 的方法，不知晓 reader 状态。两种方案：

方案 A（推荐）：在 reader 层，非 EOF 错误时不调用 `converter.Flush()`，直接返回 error 事件 + EOF。

方案 B：给 converter 增加一个 `SuppressTerminalEvents()` 方法，reader 在错误时调用。

```go
// 方案 A 实现
if err != nil && err != io.EOF {
    if !r.streamEndedWithError {
        r.streamEndedWithError = true
        errEvent := map[string]interface{}{ /* ... */ }
        r.buf.Write(sseEvent("error", mustJSON(errEvent)))
    }
    r.pendingErr = err
    continue
}
```

并在 EOF 分支中，若 `streamEndedWithError` 为 true（即先出错后排空），跳过 `Flush()`：

```go
if err == io.EOF {
    if !r.streamEndedWithError {
        flushOut, _ := r.converter.Flush()
        r.buf.Write(flushOut)
    }
    r.done = true
    // ...
}
```

**步骤 4**：配套测试：构造上游中途返回错误的流，验证客户端收到 `event: error` 且不收到 `message_stop`。

### 8. [P1] Responses 的 incomplete_reason 未细分

**问题描述**

**当前 api-gateway 实现**

**修复方案**

文件：`src/transform.go`

**步骤 1**：修改 `mapResponsesStopReason` 签名，增加 `incompleteReason` 参数：

```go
func mapResponsesStopReason(stopReason, incompleteReason string, hasToolUse bool) string {
    switch stopReason {
    case "completed":
        if hasToolUse {
            return "tool_use"
        }
        return "end_turn"
    case "incomplete":
        switch incompleteReason {
        case "max_output_tokens", "max_tokens", "":
            return "max_tokens"
        default:
            return "end_turn"
        }
    }
    return "end_turn"
}
```

**步骤 2**：更新所有调用点，传入 `incomplete_reason`：
- 非流式 [transform.go](file:///d:/Files/api-gateway/src/transform.go) `openaiResponsesToAnthropicResponse`：从 `response.incomplete_reason` 读取。
- 流式 [transform_stream.go:949-963](file:///d:/Files/api-gateway/src/transform_stream.go#L949-L963) `response.incomplete` 分支：从 `response.incomplete_reason` 读取。

**步骤 3**：grep 所有 `mapResponsesStopReason` 调用点，确保全部更新。

### 12. [P1] Responses → Anthropic 流式未处理 content_part.added/.done

**问题描述**
api-gateway 的 Responses 流式转换依赖 `response.output_text.delta` 懒开 text block，忽略 `response.content_part.added` 和 `response.content_part.done` 事件。当单个 output item 包含多个 content part（如文本 + 文件）时，block 索引错乱，客户端拼接内容错位。

**当前 api-gateway 实现**
- [transform_stream.go:797-974](file:///d:/Files/api-gateway/src/transform_stream.go#L797-L974) `openaiResponsesToAnthropicStream.Feed`：switch 无 `response.content_part.added` / `response.content_part.done` case。

**cc-switch 的做法**
- [streaming_responses.rs:205-360](file:///d:/Files/GitHub/cc-switch/src-tauri/src/proxy/providers/streaming_responses.rs#L205-L360)：显式处理 `content_part.added`（开新 block，按 `output_index`/`content_index` 索引）和 `content_part.done`（关闭 block）。

**修复方案**

文件：`src/transform_stream.go`，函数 `openaiResponsesToAnthropicStream.Feed` 的 switch

新增两个 case：

```go
case "response.content_part.added":
    s.ensureMessageStart(&output)
    partIndex := struct {
        outputIndex  int
        contentIndex int
    }{
        intFromInterface(data["output_index"]),
        intFromInterface(data["content_index"]),
    }
    // 按 (output_index, content_index) 维护 block 映射
    key := fmt.Sprintf("%d:%d", partIndex.outputIndex, partIndex.contentIndex)
    if _, exists := s.partBlockIndex[key]; !exists {
        idx := s.nextContentIndex
        s.nextContentIndex++
        s.partBlockIndex[key] = idx
        partType := "text"
        if pt := getString(getMap(data, "part"), "type"); pt == "output_text" {
            partType = "text"
        }
        output = append(output, sseEvent("content_block_start", mustJSON(map[string]interface{}{
            "type":          "content_block_start",
            "index":         idx,
            "content_block": map[string]interface{}{"type": partType, "text": ""},
        }))...)
    }

case "response.content_part.done":
    key := fmt.Sprintf("%d:%d", intFromInterface(data["output_index"]), intFromInterface(data["content_index"]))
    if idx, exists := s.partBlockIndex[key]; exists {
        output = append(output, sseEvent("content_block_stop", mustJSON(map[string]interface{}{
            "type": "content_block_stop", "index": idx,
        }))...)
        delete(s.partBlockIndex, key)
    }
```

**步骤 2**：在结构体新增 `partBlockIndex map[string]int`（key 为 `"output_index:content_index"`）。

**步骤 3**：`response.output_text.delta` 改为按当前 part 索引路由到正确的 block，而非单一 `openTextBlockIndex`。

**注意**：此修复与现有 `openTextBlockIndex` 逻辑有重叠。建议统一为 `partBlockIndex` 方案，`output_text.delta` 也走该映射。改动较大，需配套充分测试。

**若决定实施**，参考 cc-switch 的 `GeminiShadowStore`（第一轮文档 #2 已有 Go 骨架），将 `GeminiAssistantTurn` 扩展为存完整 `parts`，并在 `anthropicToGeminiRequest` 中按 session 替换。
